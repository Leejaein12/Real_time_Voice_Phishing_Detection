# KoELECTRA 보이스피싱 탐지 모델 성능 보고서 (v5)

> 작성일: 2026-04-30  
> 모델: `monologg/koelectra-base-v3-discriminator` (멀티라벨 파인튜닝)  
> 평가 스크립트: `test_scripts/run_mixed_test.py`  
> 이전 버전 대비 변경: **탐문형 피싱 데이터 500개 집중 보강** (v4 미탐 문제 해결)

## 1. 모델 구조

| 항목 | 내용 |
|------|------|
| 베이스 모델 | KoELECTRA-base-v3 (한국어 ELECTRA) |
| 분류 방식 | 멀티라벨 (Multi-label Classification) |
| 출력 레이블 | `[기관사칭, 금전요구, 개인정보]` — 3개 독립 판단 |
| 손실 함수 | BCEWithLogitsLoss (이진 교차 엔트로피, 멀티라벨용) |
| 추론 임계값 | 각 레이블 Sigmoid ≥ 0.5 시 탐지 |
| pos_weight | 전 레이블 15.0 캡 (피싱:정상 ≈ 1:10 불균형 보정) |
| 파라미터 수 | 약 110M (hidden 768, attention heads 12, layers 12) |

레이블 예시:
```
[1, 0, 0]  → 기관사칭만
[0, 1, 0]  → 금전요구만
[1, 0, 1]  → 기관사칭 + 개인정보 복합
[0, 0, 0]  → 정상
```

### 2-2. 탐문형 신규 데이터 상세 (`tamun_aug_v5`)

v4에서 미탐으로 확인된 **탐문형 피싱 수법**을 집중 보강. 직접 요구 없이 사실 확인·공감·상황 탐문으로 신뢰를 쌓는 발화 패턴.

| 라벨 조합 | 수량 | 대표 패턴 |
|-----------|-----:|----------|
| `[1, 0, 0]` 기관사칭 | 150개 | 수사기관 사칭 후 사건 연루 가능성 탐문 |
| `[1, 0, 1]` 기관사칭 + 개인정보 | 150개 | 명의도용·신분증 분실 여부 확인 유도 |
| `[1, 1, 1]` 전체 복합 | 200개 | 탐문 이후 금전·개인정보 요구로 이어지는 전체 흐름 |

탐문형 키워드 포함 비율: `"혹시"` 79% / `"있으신가요"` 56% / `"명의"` 23%  
직접 요구형 키워드 포함 비율: `"이체하"` 0% / `"주민번호"` 0% / `"OTP"` 0%

**탐문형 샘플 예시:**

> `[1, 0, 0]` — "여보세요, 000씨 계신가요? 저 서울지방경찰청 000 사무관입니다. 전화 드린 이유는 최근에 불법 금융거래와 관련해 몇 가지 확인할 사항이 있어서인데요. 잠깐 통화 가능하신가요?"

> `[1, 0, 1]` — "여보세요, 000님? 저 서울지방경찰청에서 일하고 있는 김 사무관입니다. 성매매 관련 특별법 위반 사건 때문에 연락드렸는데, 혹시 과거에 신분증이나 운전면허증을 분실하신 적 있으신가요? 정말 중요한 사건이에요."

> `[1, 1, 1]` — "[사무관]: 성매매나 불법 자금 관련해서 조사 중인데, 000씨가 어떤 경우든 신분증이나 개인정보를 타인에게 제공한 적은 없으신가요?"

### 4-2. 오탐 사례 (정상 → 피싱으로 잘못 판정)

**v5 테스트 오탐 1건 (`mixed_test_v4.json` 기준):**

| 위험도 | 판정 카테고리 | 텍스트 |
|-------:|---------------|--------|
| **96.2%** | 금전요구, 개인정보 | "지금 이체하려고 하는데 비밀번호가 생각이 안나네요. 어떻게 하면 좋을까요." |

**실제 영향 미미:** 앱의 실제 동작에서는 `analyzer.py`가 통화 전체 누적 텍스트(`_full_text`)를 KoELECTRA에 전달하므로, 앞뒤 정상 발화가 컨텍스트로 포함되어 오탐 발생 가능성이 대폭 낮아짐.

## 6. v4 미탐 해결 — 탐문형 피싱

### 6-1. v4에서 발견된 미탐 문제

`mixed_test_03.json` (KBS 공개 시나리오 포함) 테스트에서 v4 모델이 탐지하지 못한 문장:

> "본인이 개설한 적이 없다는 말씀이시죠? 그럼 혹시 예전에 제3자에게 본인 명의로 빌려줬다던가 신분증이나 운전면허증 분실한 적은 있으신가요?"  
> — KBS 공개 보이스피싱 시나리오 (2023.03), 검1선 8번 발화

| 항목 | v4 결과 |
|------|---------|
| 키워드 점수 | 60점 (필터 통과) |
| KoELECTRA 실행 | ○ |
| KoELECTRA 판정 | **정상 (위험도 ≈ 0%)** → **미탐** |
| 실제 라벨 | `[1, 0, 1]` 기관사칭 + 개인정보 |

**미탐 원인:** 학습 데이터가 "주민번호 알려주세요", "계좌로 이체하세요" 같은 직접 요구형에 치우쳐 있어, 질문·공감·상황 확인으로 신뢰를 쌓는 탐문형 수법을 정상 통화로 오인.

### 6-2. v5에서의 해결

탐문형 데이터 500개 집중 보강 후 동일 문장 재추론:

| 항목 | v4 결과 | v5 결과 |
|------|---------|---------|
| 기관사칭 | 0.01% ❌ | **100%** ✅ |
| 금전요구 | 0.00% ❌ | 56.83% ✅ |
| 개인정보 | 0.02% ❌ | **99.98%** ✅ |
| 위험도 | **0%** → 미탐 | **100%** → 탐지 |

v4에서 완전 미탐(0%)이었던 문장이 v5에서 기관사칭·개인정보 거의 확신 수준(100% / 99.98%)으로 올바르게 탐지됨.

> 금전요구 56.83%는 실제 라벨은 0이지만, 탐문형 수법이 금전 요구의 전조 단계라는 점에서 모델의 자연스러운 연관 학습 결과. 보이스피싱 앱 특성상 과탐이 미탐보다 안전.

## 7. 한계 및 향후 개선

| 한계 | 설명 |
|------|------|
| 키워드 필터 의존 | FN 94건 전부 키워드 필터 미통과 — 필터 키워드 확충이 Recall 개선의 핵심 |
| 학습 데이터 분포 | 피싱 샘플이 GPT-4o-mini 증강 기반 — 실제 사람 발화와 문체 차이 존재 |

| 신규 수법 대응 | 학습 데이터에 없는 신규 피싱 패턴은 탐지율 저하 가능성 |
| TFLite 변환 | int8 양자화 미완성 (Docker 환경에서 재시도 필요, 목표 50~100MB 이하) |
